Development tools

I use VSCode as my primary text editor. It's open-source but if you want to use it without any of Microsoft's product customization and telemetry that comes with downloading from their site, consider downloading from VSCodium instead.

I recommend installing a couple extensions to make your code writing experience more comfortable, though they're by no means required. All can be found and installed by searching their names in the Extensions Marketplace within VSCode itself (accessed via the left sidebar).

  • Live Server: Creates a local web server that live refreshes every time you save a file within your project.
  • Prettier: Formats your HTML, CSS, and Javascript + JSON for easier reading and editing (though you can do this without an extension too!)
  • leet-html: Adds highlighting and IntelliSense for HTML templates written in Javascript. If you make heavy use of Javascript for things like headers/footers/sidebars/etc. on your site, I highly recommend this one!
  • CSS Peek: Lets you quickly view info on your CSS classes and IDs by either peeking or directly opening the CSS fiie where it's defined. Helpful for quick edits and references.

My code is hosted on Github and I use deploy-to-github to automate deployments to Neocities, following this tutorial to initially set up the Github action. I also referenced one of the final sections of Alfred Baudisch's writeup on his migration to Eleventy to update my workflow file.

This one goes to Eleven(ty)

lunarlily v2.0 is now built using Eleventy (11ty), a static site generator (SSG). In essence, it makes web development more modular - instead of writing out every single page on your site manually (or by making use of reusable sections with Javascript), you can break things down into various smaller templates that you combine together, kind of like building with Lego blocks. Eleventy then generates (see? the G in SSG?) complete HTML pages that you can host on the web - like you're looking at right now! This process means that your site doesn't have to rely on things like Javascript running in-browser each time a page loads (for example, if you use a script to put the same header on every page), nor do you have to manually write (and then later update!) the same bits of code across many pages. It's pretty cool!!

Switching to an SSG let me simplify a lot of the repetitive and fiddly parts of maintaining a personal site, and also let me move away from using client-side Javascript as much as possible out of concerns about accessibility. Actually switching over was a bit of a pain (due entirely to my own incompetence and messy code from v1.0) but hopefully it's smooth sailing from here :]

I heavily relied on several guides written by other Eleventy users to guide me through the process, as well as Eleventy's own documentation. I don't know if I could've completed the migration without their help - it was quite the learning process!

  • Migrating from WordPress to 11ty (Eleventy) from Alfred Baudisch
  • Data vs. Collections in Eleventy, from Mad Baker

At some point I'd like to sit down and pen my own write-up about the experience of adopting Eleventy and all the headaches that came with the process. For now though, I'll say this: I don't necessarily recommend it for all webmasters, especially those brand new to web development! Just like with any other technology, it has its perks and drawbacks. Personally, I think trying to learn how to build a website by jumping right into an SSG would leave you with gaps in your knowledge of how it all comes together to make functioning pages - this will make troubleshooting problems a nightmare because you'll lack a solid foundation of understanding (it's a pain in the ass to figure out why something is wrong if you don't even know what it would look like when it's correct). I learned the basics of programming in college and it's something I do semi-regularly at my job, so I'm lucky to have already had experience with site building and SSGs from that. It's something to try your hand at after getting the fundamentals (HTML, CSS, site file structure, JS for dynamic content, etc.) under your belt first, in my opinion.

On the other hand, I can just as well make an argument that it's a great place to start for building a site! Say you're someone who wants to write the actual content of your site in Markdown (a very simple formatting language) and doesn't care to add too many bells and whistles to your site structure. An SSG would make it quite easy to maintain your files and remove a ton of the repetitive work that comes with building out individual web pages, all while not demanding extensive programming knowledge. Like I said - perks and drawbacks! It all depends on your usecase and scenario, and what level of technological know-how you feel comfortable with.

Due to the switch, it's a bit harder to snoop my code and understand everything going on under the hood - there's a whole bunch of important extra files and other context that you can't see in the final product hosted here! For that reason, I recommend going directly to the site repository on Github if you want to go digging, especially if you're also trying to get set up on an SSG. You can see the templates and data collections that come together to build the completed pages, giving full context to the code that makes up the final HTML pages.
